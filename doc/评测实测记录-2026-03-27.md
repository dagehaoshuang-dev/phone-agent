**过程问题分析：**

搜索阶段（~6 轮浪费）：
- 搜索框残留上次搜索词"通透堂"，`focus_and_clear_text` 清空失败（返回 0 characters erased）
- 多次尝试键盘删除键、重新聚焦输入框均未能清除
- 最终靠点击历史搜索标签"深圳按摩"绕过

排序阶段（~7 轮浪费）：
- 第 1 次点"排序"→ 坐标偏差误入店铺详情页，连续 3 次 back 返回
- 第 2 次点"排序"→ 回到了搜索页，需重新搜索
- 第 3 次点到了"分类"面板而非"排序"
- 第 4 次点到了"筛选"面板
- 第 5 次终于点对"排序"→ 选择"好评优先"成功

**根因**：大众点评筛选栏"全域/分类/排序/筛选"四按钮紧密排列，Cortex 通过 bounds 中心点计算坐标时容易偏移到相邻按钮或下方店铺卡片。

## 五、问题记录

### 5.1 mobile-use

- **极慢**：每步 2-3 次 LLM 调用（Cortex+Executor+Orchestrator），简单任务也需要数分钟
- **误入"问点仔AI"循环**：Target 定位不准导致反复点错元素，Cortex 虽然识别了重复失败模式但坐标计算仍然偏差
- **400 步递归上限**：复杂任务（排序+验证码）步骤耗尽，无法完成
- **GPT-5.4-mini 过于保守**：有时只跑几步就认为"任务受限"直接 done
- **模型配置不当导致性能浪费**：全部 Agent 用 GPT-5.4 反而比分级配置更差。优化为 Cortex 用 5.4、其余用 5.4-mini 后从 400 步超时变为成功完成
- **搜索框清空失败**：`focus_and_clear_text` 对大众点评搜索框无效（返回 0 characters erased），需依赖历史搜索标签或键盘删除键绕过
- **筛选栏坐标偏差严重**：大众点评"全域/分类/排序/筛选"四按钮紧密排列，Cortex 通过 bounds 中心点定位时频繁偏移到相邻按钮或下方店铺卡片，导致 5 次尝试才点对"排序"

### 5.2 Midscene

- **验证码天然优势**：纯视觉路线能"看到"验证码图片并操作，这是 Accessibility Tree 路线做不到的
- **速度较慢**：每次都发截图给 VLM，token 消耗大
- **YAML 脚本需要预编排**：不像其他项目一句话自动规划，需要手动拆分步骤
- **AutoGLM 适配 Bug（已修复+提 PR）**：
  - Launch 参数类型不匹配：`actions.ts` 传了字符串但 `launchParamSchema` 期望 `{uri: string}` 对象，Zod 校验失败。已修复并提交 PR（web-infra-dev/midscene#2239）
  - aiQuery 不兼容：AutoGLM 只适配了 planning 管线，insight/query 管线仍期望 `<data-json>` XML 格式，AutoGLM 输出 `<think>...<answer>` 格式。需官方适配，暂用 `ai` 指令代替 `aiQuery` 绕过

### 5.3 Open-AutoGLM

- **安全优先**：遇到验证码直接放弃，调用 Take_over 交给人工。设计理念不同
- **专用模型限制**：只能用 AutoGLM-Phone，不能切换其他模型
- **信息提取最丰富**：纯视觉路线能从截图中提取评分、评价数、距离、团购等全部可见信息

### 5.4 mobile-mcp

- **依赖决策模型质量**：Claude Opus 做决策时表现最好（主动好评排序），Qwen 做决策时会迷路
- **子 Agent 加载不稳定**：MCP 工具有时能加载有时不能，需要用脚本直接调用更可靠

## 六、结论

### 最佳方案选择

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 速度优先 | **Open-AutoGLM** | 44秒完成，9步，最快且信息最丰富 |
| 视觉+速度均衡 | **Midscene + AutoGLM-Phone** | 56秒，纯视觉，修复后可用 |
| 任务理解最准 | **mobile-mcp + Claude Opus** | 唯一主动做好评排序的 |
| 验证码场景 | **Midscene + Qwen3-VL** | 纯视觉天然优势 |
| 结构化输出 | **mobile-use + GPT-5.4-mini** | Outputter Agent 自动生成 JSON |
| 安全优先 | **Open-AutoGLM** | 遇验证码主动交人工 |

### 核心洞察

1. **专用模型 > 通用模型**：Open-AutoGLM（专用 AutoGLM-Phone）在速度和信息量上均领先，说明针对移动端训练的专用模型优势明显
2. **多 Agent 架构开销巨大**：mobile-use（5 Agent/2-3次LLM调用每步）vs Open-AutoGLM（单模型/1次调用每步），差距在 5-10 倍以上
3. **模型配置影响大于预期**：mobile-use 全用 GPT-5.4 反而超时，优化为分级配置后成功完成，说明合理分配模型比全用最强模型更重要
4. **纯视觉路线在验证码场景有独特优势**：Midscene 能"看到"并通过图标验证码，Accessibility Tree 路线做不到
5. **验证码是所有项目的硬边界**：没有任何项目能 100% 可靠地通过所有类型的验证码
6. **大众点评反爬最严格**：频繁操作触发验证码/风控，评测需要控制测试频率
7. **Midscene + AutoGLM 组合潜力大**：修复 Launch Bug 后 56 秒跑通全流程，且继承了纯视觉路线的信息密度优势。aiQuery 官方适配后将更完整
