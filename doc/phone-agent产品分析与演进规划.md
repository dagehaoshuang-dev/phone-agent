# phone-agent 产品深度分析与演进规划

> 基于源码分析 + 实测数据 + 竞品对比
> 日期：2026-03-27

---

## 一、现状评估

### 1.1 核心数据

| 指标 | 数据 |
|------|------|
| 代码量 | ~680 行 Python |
| 依赖 | 3 个（openai, uiautomator2, Pillow） |
| 架构 | 单 Agent + 工具循环 |
| 最佳耗时 | 46 秒（大众点评搜索+好评排序） |
| 最佳模型 | qwen3-max |
| 支持平台 | 仅 Android |

### 1.2 核心优势

**速度**：同等任务比 mobile-use 快 5-10 倍，比 Midscene 快 2-4 倍

**简洁**：680 行代码，3 个依赖，一个文件搞定。对比 mobile-use 5000+ 行 + 175 个依赖

**模型兼容**：OpenAI 兼容接口，已验证 Qwen/GPT/Claude 均可用

**合批操作**：一次 LLM 调用可执行多个连续操作，减少往返

### 1.3 当前问题

#### 代码层面

1. **System Prompt 规则编号混乱**：规则 7/8/9 出现了重复编号（第 101-106 行），维护困难
2. **APP_MAP 硬编码**：仅 12 个 App，新增需要改代码。微信出现两次（第 84、87 行重复）
3. **choose_model 形同虚设**：第 474-476 行永远返回 MODEL_SMART，双模型策略名存实亡
4. **验证码处理不稳定**：VLM 返回 JSON 格式不标准导致解析失败
5. **无错误恢复机制**：操作失败后没有系统性的重试/回退策略
6. **上下文压缩粗暴**：超过 12 条消息就截断，可能丢失关键操作历史
7. **截图留在上下文**：截图 base64 占大量 token，导致 Qwen 的 function calling 格式崩溃

#### 产品层面

1. **仅支持 Android**：无 iOS 支持
2. **无状态管理**：每次任务独立，无法跨任务记忆
3. **无任务进度反馈**：用户只能看到最终结果，中间过程不可控
4. **无安全机制**：没有敏感操作确认，没有操作边界限制
5. **APP_MAP 依赖手动维护**：无法自动发现已安装应用
6. **无结构化输出**：结果是纯文本，无法像 mobile-use 那样返回 JSON schema
7. **无 trace/回放**：无法查看操作历史和回放调试

## 二、产品定位分析

### 2.1 当前定位

**开发者工具/研究原型** — 一个快速验证"单 Agent 能否搞定移动自动化"的 PoC。

### 2.2 目标用户画像

| 用户类型 | 需求 | 当前满足度 |
|----------|------|-----------|
| **个人效率用户** | 一句话操控手机完成日常任务 | ⭐⭐⭐ 基本可用 |
| **测试工程师** | 自动化 UI 测试 | ⭐⭐ 缺少断言/报告 |
| **数据采集** | 从 App 批量提取信息 | ⭐⭐⭐ 跨 App 流转好 |
| **企业自动化** | 批量操作、流程编排 | ⭐ 缺少队列/调度/日志 |
| **无障碍辅助** | 帮助视障/老年人操作手机 | ⭐⭐ 需要语音交互 |

### 2.3 竞争格局

```
                    复杂度高
                      │
         mobile-use   │   Midscene
         (多Agent+重)  │   (纯视觉+跨平台)
                      │
    ──────────────────┼──────────────────
                      │
         phone-agent  │   Open-AutoGLM
         (单Agent+轻) │   (专用模型+视觉)
                      │
                    复杂度低

    速度快 ◄─────────────────────────► 速度慢
```

phone-agent 占据了**左下角（简单+快速）**的生态位。要扩大市场需要向右上移动——在保持简洁的前提下增加能力。

## 三、核心问题深度分析

### 3.1 模型依赖风险

当前最大的脆弱性：**换个模型可能就跑不了**。

实测数据：
- qwen3-max：✅ 完美（function calling 稳定）
- qwen3.5-plus：⚠️ 简单任务可用，截图上下文下 function calling 崩溃
- GPT-5.4-mini：✅ 可用但保守
- GPT-5.4：⚠️ max_tokens 参数不兼容

**根因**：不同模型的 function calling 实现差异大，phone-agent 没有做兼容层。

**建议**：
- 增加模型适配层，自动检测并适配参数差异（max_tokens vs max_completion_tokens）
- 对 function calling 输出做格式校验和修复（如字符数组拼接修复）
- 维护一个模型兼容性矩阵，测试通过的模型标记为推荐

### 3.2 UI 树 vs 截图的权衡

当前策略：**UI 树优先，截图按需**。这是正确的方向，但存在边界问题：

| 场景 | UI 树 | 截图 | 当前处理 |
|------|-------|------|----------|
| 标准 App 操作 | ✅ 足够 | 不需要 | ✅ |
| 图片内容（帖子图片） | ❌ 无信息 | ✅ 必需 | ⚠️ 需手动调用 screenshot |
| 验证码 | ❌ 看不到图 | ✅ 必需 | ⚠️ VLM 分析不稳定 |
| WebView 内容 | ❌ 部分可见 | ✅ 补充 | ❌ 未处理 |
| Canvas/游戏 | ❌ 完全不可见 | ✅ 唯一手段 | ❌ 不支持 |

**建议**：引入"智能截图"策略——当 UI 树中检测到 ImageView/WebView/验证码关键词时自动截图，不需要 LLM 主动请求。

### 3.3 Prompt 工程的可维护性

当前 System Prompt 已经有 ~30 行规则，且编号混乱（7/8/9 重复）。随着支持的 App 和场景增多，prompt 会越来越膨胀。

**问题**：
- 规则互相矛盾（"不要滑动" vs "数据不够要滑动"）
- App 特定规则（高德/小红书/大众点评）硬编码在 prompt 里
- 验证码规则写在 prompt 里但实际由代码处理

**建议**：
- 拆分为基础规则（不变）+ App 特定规则（按需注入）+ 场景规则（动态）
- 基于当前 App 包名动态加载对应的 App 操作指南
- 验证码完全由代码处理，从 prompt 中移除

### 3.4 上下文管理

当前策略：超过 12 条消息时保留前 2 条 + 摘要 + 最后 8 条。

**问题**：
- 截图 base64 留在上下文中占大量 token（一张截图 ~5-20K tokens）
- 摘要只保留 thought，丢失了具体的操作结果和屏幕状态
- Qwen 在大上下文下 function calling 格式不稳定

**建议**：
- 截图只在当前步使用，下一步立即从上下文中移除（只保留文字描述）
- 引入"工作记忆"（scratchpad），让 LLM 主动保存关键信息
- 压缩时保留操作结果（成功/失败），不只是 thought

## 四、产品演进路线

### 4.1 短期（1-2 周）：修复和加固

| 优先级 | 任务 | 影响 |
|--------|------|------|
| P0 | 修复 System Prompt 编号混乱 | 可维护性 |
| P0 | 截图后立即从上下文移除 base64 | 修复 Qwen function calling 崩溃 |
| P0 | 修复 APP_MAP 微信重复 | 代码质量 |
| P1 | 自动发现已安装 App（`pm list packages`） | 不需要手动维护 APP_MAP |
| P1 | 模型兼容层（max_tokens 自动适配） | 支持更多模型 |
| P1 | 验证码 VLM 分析 JSON 格式修复 | 验证码通过率 |
| P2 | 操作失败后自动重试（最多 2 次） | 鲁棒性 |

### 4.2 中期（1-2 月）：产品化

#### 4.2.1 结构化输出

```python
# 用户指定输出格式
agent.run(
    task="搜索附近火锅前3家",
    output_schema={"shops": [{"name": str, "rating": float, "price": int}]}
)
# 返回 JSON，不是纯文本
```

#### 4.2.2 任务编排（YAML 脚本）

```yaml
tasks:
  - name: 搜索美食
    steps:
      - action: launch
        app: 大众点评
      - action: search
        text: 深圳按摩
      - action: sort
        by: 好评
      - action: extract
        count: 3
        fields: [name, rating, price]
```

#### 4.2.3 多设备支持

- 支持同时连接多台设备
- 设备池管理
- 并行任务执行

#### 4.2.4 iOS 支持

- 通过 WebDriverAgent 支持 iOS
- 统一设备抽象层（Protocol/Interface）

#### 4.2.5 可视化回放

- 每步操作自动截图
- 生成 HTML 回放报告
- 操作时间线可视化

### 4.3 长期（3-6 月）：平台化

#### 4.3.1 MCP Server 模式

将 phone-agent 封装为 MCP Server，可被 Claude Desktop/Cursor/VS Code 等客户端直接调用：

```bash
# 启动 MCP Server
phone-agent serve --port 8080

# 任何 MCP 客户端都能连接
claude mcp add phone-agent -- phone-agent serve
```

#### 4.3.2 智能 App 适配

- 首次使用 App 时自动学习其 UI 结构
- 建立 App 操作知识库（搜索框位置、排序按钮、常见弹窗）
- 社区共享 App 适配规则

#### 4.3.3 工作流引擎

```python
# 定义可复用的工作流
workflow = Workflow("美食探店")
workflow.add_step(SearchStep(app="大众点评", query="${keyword}"))
workflow.add_step(SortStep(by="好评"))
workflow.add_step(ExtractStep(count=3, fields=["name", "rating", "price"]))
workflow.add_step(CrossAppStep(app="高德地图", action="navigate", target="${shop_name}"))

# 参数化执行
result = workflow.run(keyword="火锅")
```

#### 4.3.4 安全与合规

- 敏感操作确认（支付、删除、发送消息）
- 操作审计日志
- App 黑名单/白名单
- 速率限制（避免触发反爬）

## 五、技术架构演进

### 5.1 当前架构

```
用户指令 → [单文件 Agent] → 设备
              │
              ├── UI 树解析
              ├── LLM 推理
              ├── 操作执行
              ├── 验证码处理
              └── 上下文管理
```

### 5.2 目标架构

```
用户指令 / YAML / MCP / API
              │
    ┌─────────┴─────────┐
    │    任务管理层       │
    │  (队列/调度/状态)   │
    └─────────┬─────────┘
              │
    ┌─────────┴─────────┐
    │    Agent 核心层     │
    │  ├── Prompt 管理   │ ← App 规则库
    │  ├── LLM 适配器    │ ← 多模型兼容
    │  ├── 上下文管理     │ ← 智能压缩
    │  └── 工作记忆       │ ← scratchpad
    └─────────┬─────────┘
              │
    ┌─────────┴─────────┐
    │    设备抽象层       │
    │  ├── Android       │ ← uiautomator2
    │  ├── iOS           │ ← WebDriverAgent
    │  └── 设备池管理     │
    └─────────┬─────────┘
              │
    ┌─────────┴─────────┐
    │    能力扩展层       │
    │  ├── 验证码解决器   │ ← VLM + 人类化轨迹
    │  ├── App 知识库     │ ← 操作规则自动学习
    │  └── 安全防护       │ ← 敏感操作拦截
    └─────────────────────┘
```

### 5.3 关键设计原则

1. **保持核心简洁**：Agent 核心逻辑控制在 500 行以内
2. **能力通过插件扩展**：验证码、App 知识库、安全防护都是独立模块
3. **模型无关**：通过适配器支持任意 OpenAI 兼容模型
4. **设备无关**：通过抽象层支持 Android/iOS/模拟器
5. **单 Agent 不变**：不引入多 Agent 架构，用 prompt 工程和工具设计解决复杂性

## 六、竞品借鉴

### 从 mobile-use 借鉴

- **Outputter Agent**：任务完成后自动生成结构化输出 → phone-agent 可以在 done 时加 output_schema
- **Scratchpad 工具**：save_note/read_note 跨步骤持久化信息 → 解决截图信息丢失问题
- **Trace 录制**：`--test-name` 参数自动录制操作历史 → 调试和回放

### 从 mobile-mcp 借鉴

- **ActionableError**：错误消息附带可操作提示 → 让 LLM 自己修复错误
- **安全机制**：路径白名单、包名校验 → 防止误操作

### 从 Midscene 借鉴

- **YAML 脚本 + CLI**：声明式任务定义 → 非开发者也能用
- **HTML 回放报告**：可视化操作历史 → 调试利器
- **aiAssert 断言**：验证操作结果 → 自动化测试

### 从 Open-AutoGLM 借鉴

- **448+ App 包名映射**：大规模预配置 → 开箱即用
- **Take_over 人工接管**：遇到无法处理的场景交给人 → 安全兜底
- **Thinking 可观测**：显式输出推理过程 → 用户信任

## 七、核心结论

### phone-agent 的真正价值

不是"又一个移动自动化框架"，而是证明了一个关键命题：

> **2026 年的通用 LLM（qwen3-max/GPT-5.4），单次推理能力已经足够强，不需要多 Agent 协作就能完成复杂的手机操作任务。**

这个命题的成立意味着：
- 移动自动化的复杂度应该在**工具层**而非**AI 层**
- 好的工具设计（元素编号、合批操作、智能截图）比堆 Agent 更有效
- 速度和成本优势（5 步/46 秒 vs 400 步/超时）是实打实的

### 下一步最重要的一件事

**解决截图上下文导致的 function calling 崩溃**。这是当前最大的技术债，修复后 qwen3.5-plus 也能稳定工作，覆盖更多用户。
